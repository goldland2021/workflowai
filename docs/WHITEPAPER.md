## 1. 执行摘要

### 1.1 我们是什么

**WorkflowAI** 是面向 **机场接送、私人包车与相关地面交通服务商** 的 **AI 成交前台（AI Front Office）**。

它帮助服务商：

- 在官网 **7×24 接待**访客  
- **收集**报价所需行程与联系方式  
- 按企业规则 **建议报价**  
- 将商业决策 **升级给老板审批（Boss Inbox）**  
- 生成 **客户确认话术**，并（可选）经官方 WhatsApp（WABA）触达  

大模型负责理解与回复；**定价、升级与流程状态以结构化业务规则为准**，避免「AI 随意报价」。

### 1.2 我们不是什么

| 不是 | 原因 |
|------|------|
| WhatsApp BSP / 第二个 360dialog | 不做官方通道与 WABA 开户管道生意 |
| 通用 WATI 类全行业客服台 | 不靠群发与泛用机器人红海取胜 |
| 纯落地页工具 | 落地页是获客与验证，SaaS 本体是 AI 工作流 |

### 1.3 一句话战略

> **在接送垂直把「询价 → 报价 → 人审 → 确认」做成可订阅的软件；  
> 先自营验证与垂直复制，再抽引擎扩到相邻服务业。**

### 1.4 预期结局（务实）

- **主路径：** 小而美、可独立盈利的垂直 SaaS  
- **进取路径：** 接送 / 包车品类 AI 前台的认知品牌，并向日游、会展用车等相邻场景扩展  
- **非目标：** 通信管道帝国或泛化超级客服平台

## 2. 问题与机会

### 2.1 行业痛点

私人接送与包车多为 **小微运营商 + 高客单 + 强人工**：

| 痛点 | 表现 |
|------|------|
| 漏接 | 夜间、时差、广告高峰无人回 |
| 信息不全 | 反复追问机场、人数、行李、酒店 |
| 报价慢 | 价在老板 pen 里，员工不敢报 |
| 确认乱 | WhatsApp 复制粘贴，易错易纠纷 |
| 通道不稳 | 个人 WhatsApp 易因发送量被限制 |
| 多语言 | 英 / 繁中 / 阿语等需求上升，人力贵 |

### 2.2 市场变化带来的机会

1. **获客线上化：** Google / Meta 广告 + 多语言落地页成为标配。  
2. **用户期望即时回复：** 对标互联网体验，而非「明天回消息」。  
3. **WhatsApp 正规化：** 个人号限流推动企业走向 **WABA**，系统对接需求上升。  
4. **AI 成本下降：** 抽取字段、多语言回复在工程上已可产品化。  
5. **平台分走流量：** Klook 等吃货架；**独立站 + 自有成交链** 成为保留利润的选择。  

### 2.3 机会窗口

通用 AI 客服不懂 **等候政策、车型、分区价、人审**；  
通道商（360dialog 等）只解决 **信能发出去**。  

中间缺口是：

> **「会做接送生意的成交层」** —— 这正是 WorkflowAI 的位置。

## 3. 产品定位

### 3.1 定位陈述

| 维度 | 定义 |
|------|------|
| 品类 | 垂直行业 AI Front Office / 成交操作系统 |
| 首发行业 | 机场接送、私人包车、点对点与日包车 |
| 核心价值 | 即时接待 + 结构化收单 + 规则报价建议 + 人机协同审批 |
| 部署方式 | 云端 SaaS，网站嵌入 Widget，可选对接客户自有 WABA |
| 关系型定位 | **WABA 生态上的行业深化服务**，而非 WABA 本身 |

### 3.2 定位分层（战略地图）

```
L3  行业应用（本产品）     WorkflowAI：接送 AI 前台与审批成交
L2  通用会话平台           WATI 等（非现阶段主战场）
L1  通信通道               WABA / 360dialog / Cloud API
```

本产品站稳 **L3**；对已有 L1 的客户提供 **BYOC（自带通道）** 深化；不争夺 L1。

### 3.3 与关联资产的关系

| 资产 | 角色 |
|------|------|
| WorkflowAI（本 SaaS） | 主产品：可对外销售的软件 |
| jpairport / JP VIP Charter 等落地页 | 获客、品牌、自营验证、Demo 与案例来源 |
| 自营或合作履约 | 早期现金流与真实业务数据，非 SaaS 唯一收入 |

## 4. 解决方案与核心能力

### 4.1 端到端主闭环

```
老板注册并训练 AI 员工
        ↓
访客在网站打开聊天 Widget
        ↓
AI 解答并收集行程与联系方式
        ↓
规则引擎生成报价建议
        ↓
老板在 Inbox 通过 / 改价 / 拒绝
        ↓
生成确认内容（可选经 WABA 发出）
        ↓
跟进履约相关事项（司机、付款、改期等）
```

### 4.2 能力模块

| 模块 | 能力 | V1 重点 |
|------|------|---------|
| 账户与租户 | 注册、登录、公司隔离 | ✅ |
| Train Employee | 引导式采集业务规则 → 结构化配置 | ✅ |
| 知识库 | FAQ、政策、边界 | ✅ 基础 |
| Website Widget | 可嵌入、签名 Token、来源校验 | ✅ |
| 会话与抽取 | 多轮对话、字段提取、事件检测 | ✅ |
| 报价建议 | 基于配置规则，非模型随意定价 | ✅ |
| Boss Inbox | 人审商业决策 | ✅ |
| 预订确认 | 客户向确认文本与状态 | ✅ |
| 计费 | Stripe 订阅、Portal、Webhook | ✅ 基础 |
| WABA / 360 对接 | BYOC 收发、模板确认 | 规划 / 分期 |
| 多行业模板 | 配置包扩展 | 路线图后期 |

### 4.3 设计原则

1. **规则为源、模型为辅** —— 价格与是否升级由配置决定。  
2. **先捕获联系方式与关键字段** —— 防止访客流失。  
3. **商业决策可打断** —— 老板始终可接管。  
4. **多租户默认安全** —— 按 company 隔离数据与查询。  
5. **通道可插拔** —— Web 优先，WhatsApp/WABA 增强。

## 5. 目标用户与使用场景

### 5.1 目标客户（Buyer）

| 类型 | 特征 |
|------|------|
| 小型包车 / 接送公司 | 1～20 人，老板兼客服 |
| 多语言地接 | 服务港台、东南亚、中东、欧美旅客 |
| 已有或计划上 WABA 的运营商 | 个人号被限流后寻求正规化 |
| 有独立站或落地页的服务商 | 需把广告流量变成线索与确认 |

### 5.2 终端用户（旅客）

不直接付 SaaS 费；通过 **官网 Widget / WhatsApp** 获得更快、更清晰的询价与确认体验。

### 5.3 典型场景

1. **广告夜间进站** —— AI 收齐机场、日期、人数，留下 WhatsApp。  
2. **成田→东京 23 区询价** —— 规则给出参考价，老板一键确认。  
3. **富士山全日包车** —— 收集行程偏好，升级人工设计。  
4. **已有 360dialog 客户** —— 不换号，审批后走其 WABA 发确认。  
5. **个人号被限制** —— 官网 Widget + 表单托底，同时迁移官方号。

### 6.2 多租户模型

- 每一 **Company** 独立配置、会话、密钥与用量。  
- 应用查询 **强制 company 作用域**。  
- Widget 使用 **签名、可吊销 Token** 与允许来源校验。  

### 6.3 与获客站的关系

落地页负责 **SEO/SEM、价卡展示、品牌信任**；  
WorkflowAI 负责 **对话态成交与后台作业**。  
二者可同属运营主体，但 **SaaS 对外销售单元是 WorkflowAI**。

## 7. 与 WhatsApp / WABA 的关系

### 7.1 概念

| 术语 | 含义 |
|------|------|
| 非 WABA 模式 | 个人号或普通商业 App，常用 `wa.me` 跳转 |
| WABA | WhatsApp Business Account，官方商业账户 |
| 360dialog 等 | Meta BSP，提供 API 与号码托管 |
| 本产品 | 业务与审批层，可对接客户既有 WABA |

### 7.2 两种模式对比（用户体感）

| | 非 WABA | WABA |
|--|---------|------|
| 开通 | 极容易 | 需认证与模板 |
| 小规模人工 | 更方便 | 偏重 |
| 规模化、自动确认、接系统 | 易限流、难扩展 | **更合适** |
| 旅客发起聊天 | 都方便 | 都方便 |

### 7.3 产品策略

| 阶段 | 策略 |
|------|------|
| 默认 | **Web Widget 完整可用**，不强制 WABA |
| 客户已有 360/WABA | **BYOC 深化服务**：接 webhook 与出站确认 |
| 客户个人号被限制 | 引导正规 WABA + 官网托底，避免继续违规群发 |
| 永不作为主业 | 代开管道、转售 BSP、群发营销工具 |

### 7.4 对「是否必须上 WABA」的立场

- **业务规模化与稳定通知** → WABA 几乎是必经之路。  
- **WABA ≠ 可无限群发**；须模板合规、用户选择与质量评分。  
- 本产品在客户上 WABA 后 **价值上升**（自动确认、会话进 Inbox），但 **冷启动不阻塞于 WABA**。

## 8. 市场与竞争

### 8.1 竞争梯队

| 类型 | 代表 | 与我们的关系 |
|------|------|----------------|
| 垂直预订站 | JPC、Tokyo Airporter 等 | 争终端旅客；我们可服务「类似他们的运营商」 |
| OTA 货架 | Klook、KKday、Pelago | 分走标准化流量；我们帮独立站保利润 |
| 通用 WA 客服 | WATI、Respond.io 等 | 通道上的通用层；我们比其 **更懂接送成交** |
| 通道 / BSP | 360dialog、Twilio… | **互补**；已是其客户者恰是我们的深化对象 |
| 高端车队 | Blacklane、MK 等 | 部分自建系统；中小运营商才是主战场 |

### 8.2 差异化

1. **行业工作流内建**（字段、价规、人审、确认），非通用 chatbot。  
2. **规则定价 + AI 表达**，降低乱价风险。  
3. **Boss Inbox** 契合「老板拍板」型小企业。  
4. **BYOC WABA**，不强迫换号。  
5. **自营验证** 带来可信案例与持续迭代。  

### 8.3 竞争策略

- 不与 BSP 抢管道。  
- 不与 WATI 拼全功能清单。  
- 与 360 老客叙事：**「通道你已有，成交层找我们。」**  
- 对终端流量：用落地页 + 价卡 + 多语言建立信任，再用 WorkflowAI 承接。

## 9. 商业模式

### 9.1 收入结构（目标态）

| 类型 | 说明 | 阶段 |
|------|------|------|
| SaaS 订阅 | 按套餐 / 会话量 / 座席 | 核心长期 |
| 实施与对接 | 规则配置、WABA 对接、模板协助 | 中早期高价值 |
| 自营或履约相关毛利 | 验证与现金牛 | 早期可偏高 |
| （可选）生态合作 | 不主做消息转售 | 谨慎 |

**原则：** 早期可用交易利润养产品；中期 **订阅必须可独立解释盈利**。

### 9.2 定价方向（示意，非最终公示价）

- **Starter：** 单店、基础 Widget 与会话量  
- **Growth：** 更高用量、Inbox、多语言、WABA 连接  
- **实施包：** 一次性 onboarding  

具体数字随市场测试调整；以 **低于半个客服成本、高于纯工具心理价** 为锚。

### 9.3 单位经济关注点

- CAC（获客） vs 数月订阅回收  
- 毛利（模型成本 + 支持成本）  
- 流失率与净收入留存  
- 单客户配置工时是否可模板化  

### 9.4 独立盈利定义（建议采纳）

连续 3 个月满足其一即可宣布「SaaS 项目可独立」：

- 订阅收入覆盖 WorkflowAI 全量运营成本；或  
- 订阅 + 明确归属的实施费覆盖成本，且不依赖不可复制的一次性定制。  

自营接送利润可并列，但应 **分账看清**，避免掩盖软件是否真能卖。

## 10. 发展路线图

### 阶段 0 — 基础已具备

- 多租户、训练、Widget、会话、Inbox、Stripe 骨架  
- 垂直域模型（机场接送配置与工作流）  
- 关联落地页价卡、多语言、线索能力  

### 阶段 1 — 验证与自用打磨（约 0～6 个月）

| 目标 | 动作 |
|------|------|
| 单位经济 | 自营/合作线索：CAC、转化、单均毛利可算 |
| 产品可靠 | 延迟、抽取准确率、升级规则稳定 |
| 案例 | 可公开的响应与转化故事 |
| 通道 | 非 WABA 完整可用；WABA 规格预研 |

**退出标准：** 自己愿意每天依赖它接待，而非演示玩具。

### 阶段 2 — 垂直 SaaS 复制（约 6～18 个月）

| 目标 | 动作 |
|------|------|
| 付费租户 | 优先 10～20 家外部付费 |
| Onboarding | 行业模板，降低人手实施 |
| BYOC | 360 / Cloud API 对接 MVP（入站 + 审批出站） |
| 品牌 | 「接送 AI 前台」品类认知 |

**退出标准：** 订阅收入趋势向上，支持不拖垮团队。

### 阶段 3 — 走出单一垂直（18 个月+）

| 目标 | 动作 |
|------|------|
| 引擎复用 | 抽取 + 规则报价 + 人审 保持稳定 |
| 新行业包 | 日游、会展用车、医疗/留学接送等 |
| 叙事升级 | From airport transfer → service businesses that quote & confirm |

**不做：** 未站稳垂直前的大而全平台化。

### 明确不做（战略禁区）

- 申请成为 Meta BSP 与 360 硬刚  
- 以群发营销为卖点  
- 无付费客户前的多行业平行铺开

## 11. 技术栈与安全

### 11.1 主要技术

| 层级 | 选型 |
|------|------|

- 密钥仅服务端

## 12. 风险与对策

| 风险 | 对策 |
|------|------|
| AI 乱报价 | 规则引擎为唯一价格源；人审开关 |
| 模型成本上升 | 缓存、限流、套餐用量、可选小模型 |
| WhatsApp 政策变化 | Web 主路径不绑死；WABA 作插件 |
| 小微客户不愿付订阅 | 用自营证明 ROI；实施费 + 低档 Starter；案例说话 |
| 支持过重 | 模板化 Train、自助文档、限制过度定制 |
| 与自营品牌混淆 | 对外 SaaS 品牌统一为 WorkflowAI |
| 合规与隐私 | 最小采集、用途声明、租户数据隔离 |

## 13. 成功指标

### 13.1 产品

- Widget 会话→留下有效联系方式的比率  
- 关键齐报价字段的比率  
- 建议价→老板处理时延  
- 抽取准确率（机场、日期、人数等关键字段）  

### 13.2 商业

- 付费公司数、MRR、流失率  
- CAC 回收周期  
- 自营线索转化（验证期）  
- 支持工时 / 租户  

### 13.3 里程碑建议

| 里程碑 | 标志 |
|--------|------|
| M1 可用 | 自用稳定接待 |
| M2 可卖 | ≥1 个外部付费 |
| M3 可独立 | 订阅覆盖软件成本（见 9.4） |
| M4 可扩展 | ≥10 付费 + WABA BYOC 上线 |
| M5 出垂直 | 第二行业 ≥3 付费客户 |

---

## 14. 结语

WorkflowAI 要解决的不是「再做一个聊天框」，而是：

> **让小型接送与包车企业，在数字获客时代，仍然买得起一个「懂行的前台」——  
> 即时响应、收齐信息、按规则建议价格、关键决策留给老板，  
> 并在需要时接到他们已经拥有的官方 WhatsApp（WABA）上。**

通信管道属于 Meta 与 BSP；  
通用客服红海属于 WATI 们；  
**成交垂直与人机协同工作流，属于我们。**

前景判断：**适合做成可独立盈利的垂直 SaaS，并以行业模板谨慎外溢；  
不适合以成为 360dialog 或泛化超级平台为第一目标。**
